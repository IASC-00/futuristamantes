**Security:**
- Static site — no server-side attack surface. Main risks are at the host/CDN layer.
- No security headers can be set from HTML alone. The deploy host (Cloudflare Pages / GitHub Pages / Render Static) must set: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`.
- Formspree is a third-party form processor — data leaves the site. Note in privacy section if/when a privacy policy is added.
- Google Fonts loaded from `fonts.googleapis.com` — Google receives visitor IP + UA on every page load. EU visitors: this is a legitimate GDPR touch point under Schrems II case law. For a US-focused studio probably fine; for EU clients, self-host the fonts.

### F.2 Board member / governance

- Brand positioning is clear and differentiated: "cinematic urban futures studio" is a real category that doesn't yet have a dominant player. That's the opportunity.
- The site currently has zero proof-of-work. Three field studies are named but none are accessible. The Archive section (Sun Ra, Jane Jacobs, Graffiti...) establishes intellectual lineage but substitutes for portfolio. A board would want: one complete, fully realized field study visible before launch.
- The application form is asking for budget ranges ($10K–$150K+) but there's no credentialing on the site that justifies those numbers. Boards call this "the ask-credibility gap."
- Governance risk: the studio is not incorporated, there's no "About the team" section, and there's no contact information beyond the form. For institutional clients (city agencies, anchor institutions), this is a trust barrier.
- Recommendation: add one "Proof of Work" page — a single completed field study with full visual treatment — before any paid outreach. It de-risks the brand instantly.

### F.5 MIT-trained engineer / technical

- HTML is semantically correct: `<nav role="navigation">`, `<article>` for field cards, `aria-label` on all major sections, `aria-hidden` on decorative elements. This is notably better than most portfolio sites.
- `loading="lazy"` on all non-hero images — correct.
- `preload="metadata"` on the hero video — correct (avoids auto-downloading the full video on mobile).
- `playsinline` on video — required for iOS autoplay compliance.
- `novalidate` on the form with JS-side validation is acceptable, but the current JS doesn't appear to have client-side field validation — just a submit handler. Should add basic `required` field check before the fetch fires.
- Formspree integration: the static form action URL (`action="https://formspree.io/f/placeholder"`) is a fallback for no-JS environments — progressive enhancement is correct. The JS handler (`fetch`) takes over when JS is available. Good pattern.
- No service worker / offline capability — not needed for this site type.
- Missing: `<link rel="canonical" href="https://futuristamantes.com" />` — add before indexing.

### F.7 Privacy / data lawyer

- The application form collects: future vision, problem statement, why now, stakes, budget range, services sought. This is pre-sales data — no PII beyond potentially what's included in free-text responses.
- No name or email field on the apply form. This is unusual (and actually privacy-forward) — applicants can't be identified unless they self-identify in free-text.
- Formspree stores submitted data on their servers. Add a one-line disclosure below the form: "Your submission is processed by Formspree. See formspree.io/legal."
- No cookie banner needed — no cookies set, no analytics, no tracking pixels visible.
- No privacy policy linked anywhere — not required for a simple static site with no account creation, but should be added before any paid client engagement that involves exchanging PII.
- Google Fonts: visitor IPs transmitted to Google. EU-accessible site should self-host fonts or use `font-display: swap` with a local fallback only.

### F.8 DevOps / SRE

- Static site — no backend, no database, no workers. Operational complexity is near zero.
- **Recommended host:** Cloudflare Pages (free, private repo supported via direct upload, CDN-native, custom security headers configurable via `_headers` file). GitHub Pages is free but requires public repo.
- `_headers` file for Cloudflare Pages would set all required security headers in one place — no server code needed.
- Uptime: static CDN sites have 99.99%+ SLA with any major host. No monitoring needed at this scale.
- Deploy pipeline: push to `main` branch → auto-deploy via Cloudflare Pages or GitHub Actions. Zero config once connected.
- No build step required — pure static. Direct file deploy, no Node.js build process.
- Risk: the live futuristamantes.com WordPress site is on LiteSpeed hosting. If/when replacing it with this static rebuild, DNS cutover must happen atomically. Steps: (1) confirm new host is serving correctly at staging subdomain, (2) lower TTL on DNS 24h before cutover, (3) flip A/CNAME, (4) verify redirect from www → apex or vice versa.
